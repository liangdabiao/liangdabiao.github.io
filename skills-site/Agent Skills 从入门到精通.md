## §01 一场 2850 亿美元的恐慌

### 01.1 2026 年 2 月 5 日，那条新闻

2026 年 2 月 5 日，一个普通的星期四。

Anthropic 没有发新模型，也没宣布什么算法突破。它给 Claude 加了一项新能力：Agent Skills，也就是"技能包"。

然后，纳斯达克软件板块单日蒸发了约 2850 亿美元。高盛的软件股组合跌了 6%，法律科技公司 LegalZoom 暴跌 20%，汤森路透跌了 16%。业内人把这天叫"SaaS Apocalypse"——SaaS 末日。

我第一次看到这新闻，反应和很多人一样：就这？不就是多了个功能吗？

直到我认真研究了背后的逻辑，才意识到资本市场的恐慌是有道理的。

> **标叔的经验**：市场的反应，往往比我们的直觉快半拍。
>
> 当时我也觉得"不就是个插件"？后来我亲手用了一个合同审查 Skill，才发现它绕过了我过去要点的七八次鼠标。那一刻我才懂，恐慌的不是功能，是"入口"没了。

### 01.2 界面即阻碍：SaaS 的锁被撬开了

要理解这件事，得先回到一个问题：软件公司靠什么赚钱？

答案不是技术，也不是数据，是入口。

传统 SaaS 是一套"三明治结构"：底下是数据，中间是业务逻辑，最上面是一层图形界面（GUI）。你要用软件，就得学这套界面——怎么上传、怎么配置、怎么导出。

这套界面看着是"方便用户"，本质是一道锁。你学会了 A 家的界面，换 B 家要重新学。你的数据在 A 家存了三年，迁移成本高得吓人。

Agent Skills 出现后，这逻辑被从根上断掉了。

举一个具体例子。一家公司要审一份采购合同，看有没有违规条款。

传统 SaaS 流程：买法律科技软件 → 员工学怎么用 → 手动上传合同 → 配置审查规则 → 等系统跑 → 手动复核 → 导出报告。人得参与好几个环节，软件的界面是绕不开的中转站。

加载了 Legal Skill 的 Agent：直接访问企业云盘里的合同 → 按 Skill 内置的法律 SOP 自动比对风险 → 直接输出合规报告。用户全程不用打开任何界面，不用学任何操作。

UI，被整个绕过去了。

黄仁勋的判断是："AI 不会消灭 SaaS，SaaS 只是变成了基础设施。"这句话给 SaaS 留了面子，但"基础设施"意味着什么——水电煤也是基础设施，可没人说自来水公司"掌握用户入口"。

Agent Skills 消灭的不是软件，而是软件的独立入口权。

### 01.3 三个认知：Tool、Workflow、Skill 不是一回事

很多人看完上面的描述，第一反应是：这不就是 API 吗？这不就是 Workflow 吗？

不是。这里有一个必须厘清的认知误区。

| 概念 | 它是什么 | 类比 | 标叔的结论 |
|------|----------|------|-----------|
| Tool | 单一能力，只会做一件事 | 一把锤子 | 太碎，拼不出活 |
| Workflow | 把 A→B→C 串起来，但流程写死 | 一张施工图 | 太死，改不了路 |
| MCP | 通信协议，解决 Agent 和工具怎么"说话" | 插座 | 只接电，不教活 |
| Skill | 可复用的业务执行单元 | 有经验的工人 | 真正能干活的 |

Tool 是一把锤子。Workflow 是一张施工图。Skill 是一个有经验的工人——他知道什么时候用锤子，什么时候用电钻，出了问题怎么处理，还记得上次这类活儿怎么干的。

一个完整的 Skill，包含六样东西：Prompt 策略（怎么理解任务）、Tool 调用链（用什么工具、按什么顺序）、SOP 流程（业务标准作业程序）、Memory 机制（记住什么、忘掉什么）、权限控制（能访问什么）、失败恢复（出错了怎么办）。

有人把 Skill 的标准形态 skill.yaml 比作 Docker 的 Dockerfile。我觉得很准。Dockerfile 定义了一个容器环境，让代码能在任何地方跑；skill.yaml 定义了一套业务能力，让 Agent 能在任何地方调用它。

### 01.4 标叔的结论：Skill 是 2026 年最该懂的词

先给结论：如果你今年只想搞懂一个 AI 概念，就选 Agent Skills。

它不是又一个花哨词。它改的是软件的底层逻辑——从"人学界面"变成"Agent 直接拿结果"。

这个变化，对写代码的人、做产品的人、做业务的人，全都相关。

> **核心建议**：别急着背定义。
>
> 先记住一句话：Skill 是把"专家怎么干活"打包成 Agent 能直接调用的能力单元。后面所有章节，都是这句话的展开。

装完了？故事听完了？下一章，你自己跑一个。

## §02 跑通你的第一个技能

### 02.1 三种入口：Claude Code / OpenClaw / Codex

2026 年，你至少有三个地方能跑 Skill。

| 入口 | 谁家的 | 适合谁 | 标叔的结论 |
|------|--------|--------|-----------|
| Claude Code | Anthropic | 写代码、做工程 | 工程场景最稳 |
| OpenClaw | 开源社区 | 想要"万物皆技能"的极简框架 | 生态最野 |
| Codex | OpenAI | 终端/IDE/桌面全形态 | 工作流最全 |
| 扣子 / Workbuddy | 云端（字节 / 创业团队） | 不想装环境、网页里直接用 | 零门槛入门 |

前三个是本地智能体，装在你电脑上跑；后两个是云端智能体，技能存在厂商服务器上，你只管在网页里点。新手如果怕配环境，先用扣子或 Manus 感受一下"加载技能→说句话→出结果"的节奏也行。

Claude Code 是命令行式的 AI 编程助手，对上下文控制得很准。OpenClaw 是个"空壳"框架——核心只负责调度，所有能力都外挂成 Skill，它的技能仓库叫 ClawHub。Codex 在 2026 年 2 月发布 GPT-5.3-Codex 后，从代码补全工具升级成了横跨终端、IDE、桌面、浏览器和电脑屏幕的 Agent 系统。

> **注意**：新手别纠结"选哪个最好"。
>
> 三个本地入口都能跑 Skill，核心机制一样。先挑你用得最顺手的那个上手，跑通比选型重要。云端入口则适合先零成本体验。

### 02.2 三分钟装好第一个 Skill

我用 OpenClaw 举例，因为它最直观。

第一步：装好 OpenClaw 客户端。

第二步：去 ClawHub（clawhub.ai）搜一个你用得上的技能。比如"会议纪要"技能。

第三步：点挂载。命令大概长这样：

```bash
# 从 ClawHub 拉取并挂载一个技能包
openclaw skill install meeting-minutes
# 挂载后，下次你说"整理一下刚才的会"，它自动加载
```

预期结果：终端显示"Skill loaded"。从此你说"帮我整理会议纪要"，Agent 不再要你解释格式，直接按技能里的模板产出。

如果你用 Claude Code，更简单——把技能文件夹放进项目的 `.claude/skills/` 目录，重启会话就行。Anthropic 在 2025 年 10 月就开源了官方技能仓库 `anthropics/skills`，里面十几个示例技能，下载即用。

> **标叔的经验**：不会写技能也能先用起来。
>
> 现在有不少"技能创建器"类工具（比如 skill-creator），你只需要说"我要一个每周自动总结 RSS 的技能"，它会帮你把 SKILL.md 搭出来，甚至主动补上你没想到的检查清单。你先当使用者，再当创作者，路径更顺。

### 02.3 第一次"不解释就干活"的体验

我第一次跑通，是在给团队整理周会纪要。

以前我要说一大段："按这个模板，先写结论，再列待办，@到人，最后标优先级。"每次都得重复。

装了会议纪要 Skill 之后，我只说了一句："整理刚才的会。"它直接出了带结论、待办、责任人和优先级的纪要。我没解释任何格式。

那一刻我真正理解了 Skill 的价值：它把一次做出来的经验，存成了未来的能力。

> **标叔的经验**：判断一个 Skill 好不好，看"你还要解释几遍"。
>
> 好的 Skill，你说一遍需求它就懂。差的，你每次都得把规矩从头讲。差别就在 description 写没写对，这个我们 §03 细讲。

跑通了？感觉还行？下一章，我们拆开看它内部到底长啥样。

## §03 看门道：Skill 到底长什么样

### 03.1 一个文件夹，一个文件

很多人以为 Skill 是个超长 Prompt。错了。

Anthropic 对 Skills 的定义很简单：Skills 是一组组织好的文件，用来打包可复用的流程知识。说白了，它就是一个文件夹。

这个设计是故意做得很简单。它简单到任何人、任何 Agent，只要有一台电脑，就能创建和使用。你可以把它放进 Git 做版本管理，可以塞进网盘，也可以压缩成 zip 发给团队。

最早的误解来自"自动总结"。很多人第一次做 Skill，会把一段成功对话丢给 AI："请总结成一个可复用的 Skill。" AI 吐出来一篇长 Prompt，看着完整，下次复用却漏步骤、混规则、乱输出。

原因不是 AI 不会写文档，是它不会自动做结构设计。

一个最简单的技能，里面就一个文件：`SKILL.md`。你甚至不需要任何脚本或参考文档，先跑通再说。等后面需求变复杂，再往里加 `references/`、`scripts/`、`assets/`。

### 03.2 SKILL.md 的三个部分

打开任意一个 `SKILL.md`，你会看到大致如下的结构：

```markdown
---
name: meeting-minutes
description: 当用户需要把一段会议录音或聊天记录，整理成含结论、待办、责任人和优先级的纪要时使用。
---

# 会议纪要 Skill

## 流程
1. 读取输入的会议内容
2. 按结论 / 待办 / 责任人 / 优先级四块组织
3. 待办必须 @ 到具体人
...

## 检查清单
- [ ] 是否遗漏了任何决策点？
- [ ] 待办是否都指定了责任人？
```

它分成三个部分，各有各的用处：

| 部分 | 位置 | 干什么用 | 标叔的结论 |
|------|------|----------|-----------|
| name | 第一个 `---` 上方/内 | 技能 ID，也是文件夹名 | 让 Agent 区分彼此 |
| description | 同上，YAML 里 | 一句话说清"干什么、何时用" | 触发命门，§03.3 细讲 |
| 正文 | 第二个 `---` 之后 | 具体步骤、规则、检查清单 | 被选中才读 |

两个 `---` 之间的部分有个专门的名字，叫 **YAML 前置信息（YAML frontmatter）**。你不用抠每个字的意思，把它看成技能的"名片"就行。

一个关键机制：智能体启动时**不会立刻读完整正文**，而是只扫描每张名片（name + description），用来判断当前任务该调哪个技能。只有判断要用，才去读正文里的详细指令。

> **标叔的经验**：创建器比你想的周到。
>
> 你创建技能时没提"检查清单"，skill-creator 这类工具会根据需求自动补上——它判断哪些环节容易出错，主动加检查项。你不用事事交代，让工具帮你想到。

### 03.3 description 不是广告，是路由触发器

这是最容易踩的坑，也是 Skill 质量的命门。

很多人把 description 写成："这是一个强大的、多功能的智能信息处理 Skill。"漂亮，但触发信号极弱。模型看了不知道什么时候该用它。

好的 description 应该描述"用户什么时候需要它"，最好来自真实用户原话：

```text
# 差的写法（这是广告）
这是一个强大的会议纪要生成技能。

# 好的写法（这是路由条件）
当用户需要把一段会议录音或聊天记录，整理成含结论、待办、责任人和优先级的纪要时使用。
```

前者是广告，后者是 Agent 的判断条件。一个告诉模型"我多厉害"，一个告诉模型"什么时候轮到我上场"。

> **核心建议**：写 description 时，先假装你是模型。
>
> 你手里有一百个 Skill，用户说一句话，你得挑一个。你靠什么挑？靠每个 Skill 的 description 里写的触发场景。所以写得越像"用户原话"，越容易被正确加载。

### 03.4 技能存放在哪里

技能跑通了，它到底躺在硬盘的哪个角落？这关系到你能不能复用、共享、备份。

**云端智能体**（扣子、Manus 这类）

技能存在厂商的服务器上。你只用在网页界面里点"添加/管理"，不用管路径。好处是零维护，坏处是它不跟着你的电脑走。

**本地智能体**（以 Claude Code 为例）

按作用范围分两层：

| 层级 | 路径 | 跟着谁走 | 标叔的结论 |
|------|------|----------|-----------|
| 全局技能 | `~/.claude/skills/` | 电脑 | 你的私有百宝箱，任何项目都能用 |
| 项目技能 | `<项目>/.claude/skills/` | 项目 | 存进代码库，团队直接共享 |

全局技能也叫"个人技能"，相当于你的私有百宝箱，整台电脑生效。项目技能存在当前项目里，只在那个项目生效；因为它在代码库里，同事拉下来就能用。用 skill-creator 这类命令行工具建的技能，默认通常落在全局目录。

**重要：技能文件格式是通用的**

不同智能体的存放路径略有不同。Claude Code 用自己的 `.claude/skills/`，而大多数采纳 **Agent Skills 开放标准**的智能体（如 OpenCode 等）统一用 `.agents/skills/`。路径不同，但 `SKILL.md` 的格式完全一样——你在 Claude Code 写的技能，复制到 OpenCode 目录下就能直接用，反过来也一样。这就是为什么技能能跨平台流动。

> **注意**：你不用死记路径。
>
> 想不起来时，直接问你的智能体："我的技能装在哪里？"它会告诉你。

### 03.5 SKILL.md 是入口，references/ 和 scripts/ 是后盾

一个好 Skill 长大了，通常长这样：

```text
my-skill/
├── SKILL.md          # 入口：什么时候用、先做什么、后做什么
├── references/       # 参考资料：模板、配置、示例、术语表
├── scripts/          # 脚本：抓数据、解析、排序、去重等稳定动作
└── assets/           # 素材：字体、主题、版式骨架
```

| 位置 | 应该放什么 | 不应该放什么 | 标叔的结论 |
|------|-----------|-------------|-----------|
| SKILL.md | 触发条件、核心流程、边界规则 | 大段模板、长背景 | 越短越准 |
| references/ | 配置、模板、示例、输出契约 | 每次必执行的步骤 | 按需读 |
| scripts/ | 解析、排序、去重、校验 | 需要价值判断的活 | 交给机器 |
| assets/ | 字体、主题、版式 | 会变的逻辑 | 保持稳定 |

关键点：复杂 Skill 不怕内容多，怕的是把内容一次性塞给模型。文件系统本身就是一种上下文工程——模型只在需要时，才去读对应的文件。

### 03.6 渐进式加载：模型不会一次读完所有技能

Anthropic 采用了一个很朴素但关键的设计：运行时，模型先只看到每个 Skill 的元数据（名字 + 描述）——也就是 §03.2 说的那张"名片"。

它知道自己有这个 Skill，但不会立刻把整个 SKILL.md 读进来。等判断当前任务需要了，才去加载完整内容。

这样做的好处只有一个，但足够重要：同一个 Agent 可以拥有几百、几千个 Skill，却不会一开始就把上下文窗口占满。它只在处理具体任务时，动态把相关 Skill 拉进来。

这也解释了为什么"把所有能力塞进一个大 Agent"不是好方向。大而全的框架会把工具定义、协议细节、长文档塞满上下文，换来更高延迟、更高成本和更多误用。薄驾驭、厚技能，才是正路——这个我们 §08 专门讲。

看懂了形态、也知道它躺在哪里了？下一章，我带你钻到表层底下，看清楚智能体到底由什么组成、技能在里头算哪一块。

## §04 看门道：智能体的六大要素

### 04.1 先分清：聊天机器人 ≠ 智能体

在拆技能之前，得先搞清一个更底层的问题：你天天用的"AI"，到底是个啥？

最朴素的区别：

- **AI 聊天机器人**：像个"美食顾问"。你问番茄炒蛋怎么做，他背得出完美菜谱。但他没有厨房，切菜颠勺的活儿还得你自己干。
- **智能体（Agent）**：像个"由厨师坐镇的完整厨房"。你说"做番茄炒蛋"，厨师会动用厨房设施，把菜切好炒好，直接端到你面前。

用两个极简公式一对比就清楚了：

```text
AI 聊天机器人 = 大模型 + 上下文        （有脑子有记忆，但只能纸上谈兵）
智能体       = 大模型 + 上下文 + 工具 + 技能  （配齐设施，能真正动手）
```

行业里最权威的定义来自翁荔（Lilian Weng）：`agent = LLM + memory + planning + tool use`。本书的公式和它在工程落地层面对应——上下文对应短期记忆（范围更广），技能则把"规划能力"固化成了可复用的操作说明。

公式里只列了四大块。要真正转起来，还差一头一尾两个外围要素：触发干活的**提示词**，和接外部工具的**MCP（通用插座）**。集齐，就是六大要素。

### 04.2 AI 厨房：用一间厨房记住六要素

抽象概念不好记，我用一间"AI 厨房"打比方。你（用户）是厨房主人兼总指挥，下指令、定标准、验收。核心处理中枢是"AI 厨师"（大模型），自带常识，负责理解和关键判断。

| 要素 | 厨房里的角色 | 一句话 |
|------|------------|--------|
| 提示词 | 你对厨师下的指令 | 告诉 AI 做什么 |
| 大模型 | 厨师（大脑） | 理解、推理、下判断 |
| 上下文 | 厨房台面 | 当下要用的东西都摆这 |
| 工具 | 刀铲锅具 | 让厨师能动手 |
| 技能 | 菜谱 | 教厨师怎么做 |
| MCP | 通用插座 | 让外部电器接进来 |

打比方是为了好记，不追求和代码一一对应。抓住"足够像"的特征就行。下面六大块逐一拆。

### 04.3 提示词：你对厨师下的指令

提示词（prompt）就是你跟 AI 说的话——核心是"告知做什么"。

它最大的特点是**临时性**：只在当前对话生效。厨师不会自动记住你的口味，明天想吃同样的菜，你得重新交代一遍。它能下达个性化要求，但不具备跨对话自动生效的能力。

> **标叔的经验**：技能其实在"抵消提示词的临时性"。
>
> 你要的规矩写成 Skill 之后，下次不用重新交代。这就是技能比单次提示词高级的地方——把临时指令变成常驻能力。

### 04.4 大模型：厨师（也是大脑）

大模型就是那位真正动手的厨师，是系统的"大脑"与执行者。但脱离厨房，它只是个极聪明的"文字生成器"——能对答如流，却什么都做不了。

当工具、技能这套"硬件"配齐，大模型就解锁了智能体最核心的能力：**执行层面的自主决策**。你只说"吃什么"（目标），厨师自己规划步骤、决定先看哪份菜谱、先切菜还是先打蛋。你不用像提线木偶一样每步发一条指令。

吃什么你说了算，怎么做厨师借助设施和菜谱替你搞定。

### 04.5 上下文：厨房台面（有限空间）

厨师下厨离不开台面。食材、调料、菜谱、盘子，当下要用的都摆这。台面面积有限，摆太满就无从下手；得把早先看过的菜谱挪走腾空间，但关键信息被移走，厨师就可能"断片儿"，该放盐时放了糖。

**上下文（context）就是台面上堆的所有东西，上下文窗口（context window）就是台面的容量上限。** 你跟 AI 的对话、当前加载的技能名片、工具返回的结果，都临时放在这块"工作记忆"里。空间满了，早先的内容会被挤掉，模型就开始"忘事"。

这一点**直接影响技能设计**：技能不能一股脑堆进上下文，而该"按需加载"——启动只加载名片，用到才读正文。这就是 §03.6 那套机制的底层原因。

### 04.6 工具：厨房用具（内置 / 扩展）

菜刀、案板、炒锅——厨师做饭的用具。对大模型来说，"读文件""生成图片""搜网页"本质都靠调用工具（tool）。工具分两类：

- **内置工具**：厨房自带的刀铲，智能体原生具备（读写文件、执行代码、搜网页）。
- **扩展工具**：要插电的烤箱、外接搅拌机，得通过特定接口（插件、API、协议）才能接（操作浏览器、连数据库、对接飞书）。

没有工具的智能体，就像没有用具的厨房——厨师厨艺再好也巧妇难为无米之炊。

### 04.7 技能：菜谱（以及框架和模型的分工）

你把"番茄炒蛋"的做法写成标准菜谱，存起来。以后做这道菜，厨师直接看菜谱，不用你每次口述，换个厨师照着做也行。

**技能（skill）就是写给厨师看的菜谱，是给大模型用的操作手册。** Anthropic 的官方定义："技能是一组有组织的文件集合，为智能体打包可组合的流程性知识。"

拆开看那句话：

- "有组织的文件集合" → 一个文件夹，核心是 `SKILL.md`（对应做法说明），复杂时可含脚本、参考文档、静态资源（像菜谱附带采购清单、摆盘图）。
- "可组合" → 技能各自独立，需要时能组合使用。
- "流程性知识" → 装的是"怎么把一件事做好"（流程、规范、检查清单），不是百科全书。

这里有个写技能必须搞清的点：**技能到底给谁用？** 大多数情况你把"大模型"和"智能体"当整体就行。但真要写技能，得区分两个角色：

- **大模型（大脑）**：负责思考、推理、下指令。
- **智能体框架（harness）**：负责加载技能、管理上下文、调度工具的底层程序。Claude Code、OpenClaw、Codex、Cursor、VS Code 里的 Copilot，这些都是不同的框架。

调用技能时，两者分工明确：

| 谁 | 干什么 | 怎么干 |
|----|--------|--------|
| 框架 | 递本子、跑代码 | 把 `SKILL.md` 和参考文档递给模型；脚本（如 Python）由框架直接在本地跑，**不经过模型** |
| 模型 | 读本子、下指令 | 读 `SKILL.md` 和参考文档，弄懂步骤后，输出一条指令让框架触发脚本 |

一句话：模型出主意，框架提供行动支持。同一个技能文件夹装进不同框架都能用，正因为 Agent Skills 开放标准规定了框架怎么加载和执行技能。这层分工，正是 §05 把"稳定动作脚本化"、§08 讲"薄驾驭厚技能"的底层依据。

### 04.8 MCP：通用插座

厨房若每种电器都有专属插头，就乱套了。你的智能体要对接数据库、邮件、日历、飞书、钉钉……各家用法不同，单独适配也乱套。

**MCP（模型上下文协议）就是 AI 世界的"通用插座"。** 对工具厂商：把插头做成通用款，就能供厨房用。对智能体：装了通用插座，所有带通用插头的设备直接接入。

一句话厘清三者关系：**工具让智能体能动手，技能教智能体具体怎么做，MCP 让各种外部工具无缝接入。** 本书只需知道 MCP 的角色即可，细节不展开。

### 04.9 一张表记住六大要素

| 要素 | 厨房角色 | 干什么 | 对造技能的影响 |
|------|----------|--------|--------------|
| 提示词 | 你下的指令 | 告知做什么（临时） | 技能把临时指令变常驻 |
| 大模型 | 厨师 | 理解、判断、表达 | 把判断留给模型 |
| 上下文 | 厨房台面 | 当下工作记忆（有限） | 必须按需加载，别撑爆 |
| 工具 | 刀铲锅具 | 动手能力 | 稳定动作下沉给工具/脚本 |
| 技能 | 菜谱 | 怎么干的操作手册 | 你主要造的就是它 |
| MCP | 通用插座 | 接外部世界 | 让技能够得着外部系统 |

机制摸清了。下一章，我教你从零造一个技能——前面讲的"上下文有限""脚本归框架跑""名片负责路由"，会全部变成你写 Skill 的具体动作。

## Part 2: 核心能力

深入 Skill 的关键能力。每章一个核心动作，读完你能自己造、自己串。

## §05 从任务到技能五步走

### 05.1 先给结论：别让 AI 总结，要自己拆

我见过太多人，第一次做 Skill 就把成功对话丢给 AI："请总结成一个 Skill。"结果得到一个长 Prompt。

真正好用的 Skill，不是"把上次成功经验写得更详细"，而是先拆 workflow（工作流）。

我用 Hermes-rss 这个真实技能举例。它的目标很简单：读一批 RSS 源，筛选值得看的，按固定格式输出摘要。

如果让 AI 自动总结，很容易写成一段 Prompt："请读取这些 RSS 源，解析 XML，去重，排序，筛选，按我喜欢的格式输出。"这段话不算错，但它不是一个好 Skill——因为里面混了三种不同性质的任务。

### 05.2 五步拆法

第一步：写清楚触发场景。回答一句话："用户在什么情况下需要这个 Skill？"这直接影响 description。不要写"用于处理信息"，要写"当用户需要从多个 RSS 源筛选文章并输出阅读摘要时使用"。触发越清楚，越不容易乱用。

第二步：拆出固定流程。把流程写成顺序动作：读配置 → 跑脚本 → 拿结构化列表 → 按偏好筛选 → 按模板输出。SKILL.md 里只写模型需要知道的路线，太底层的细节如果可以脚本化，就别塞进流程。

第三步：把材料移到 references。凡是"需要参考但不适合写进入口"的内容——输出模板、风格偏好、配置说明、示例——都先放到 references/。入口文件更短，材料也更好维护。

第四步：把稳定动作脚本化。问自己三个问题：这动作有明确规则吗？每次都该得到稳定结果吗？让模型做反而容易出错吗？如果大多是"是"，就脚本化。解析 XML、去重、排序，都不该让模型每次扮演解析器。

```python
# scripts/rss_reader.py —— 稳定的机械活交给脚本（由框架直接跑，不经过模型）
import feedparser
feeds = ["https://a.com/rss", "https://b.com/rss"]
# 抓、解析、去重、排序，输出结构化结果
for url in feeds:
    parse_and_collect(url)
# 模型只负责后面"哪条值得读"的判断
```

第五步：把判断留给 AI。AI 的价值不在替你做机械活，而在处理规则写不死的地方——这篇文章值不值得读，归到技术还是商业，摘要怎么写才清楚。把这些留给 AI，Skill 才又稳又活。

### 05.3 拆完之后的样子

```text
Hermes-rss/
├── SKILL.md              # 只写：触发、顺序、边界
├── scripts/
│   └── rss_reader.py     # 抓、解析、去重、排序
├── references/
│   ├── config.yaml       # RSS 源、抓取数量、过滤词
│   ├── output-format.md  # 最终输出模板
│   └── workflow-contract.md # 脚本输出字段，AI 怎么用
```

这样拆完，Skill 才从"长 Prompt"变成"可复用 workflow"。这正好对应 §04.7 说的分工：脚本归框架跑，判断归模型。

> **标叔的经验**：写完别只问"能不能跑"。
>
> 用这份清单自查：description 说清触发场景了吗？SKILL.md 只写入口流程了吗？有能移到 references 的模板吗？有能脚本化的稳定动作吗？换一次输入，它还能按同样路径执行吗？一半答不上来，它就还是个长 Prompt。

步法学完了。下一章，我直接给你三个能抄的。

## §06 三个真实可用的 Skill

### 06.1 会议纪要 Skill

这是最容易上手、回报最高的一个。

触发场景写清楚："当用户需要把会议录音、转录或聊天记录，整理成含结论、待办、责任人和优先级的纪要时使用。"

结构很简单：

```text
meeting-minutes/
├── SKILL.md          # 触发 + 输出结构
├── references/
│   └── format.md     # 纪要模板：结论/待办/@人/优先级
└── scripts/          # 可选：把录音转文字的脚本
```

核心价值：你不用再每次解释"纪要要长什么样"。一句话，它出标准件。

### 06.2 写作风格 Skill

这个技能，是把自己的文风固化下来。

很多团队要统一文档风格。你把自己的偏好写进 references/：用词习惯、段落长度、禁用句式、语气。下次让 Agent 写东西，它先读这个风格文件。

我用它管公众号稿子。我的风格文件里写着：短句为主、第一人称、不用"强大的"这种空词、转折用"但"不用"然而"。Agent 写初稿就贴着我的人设，改起来省一半劲。

> **核心建议**：写作风格 Skill 是最好的"个人资产"。
>
> 它不只能给你自己用，还能发给团队，让所有人的对外文档保持同一种声音。

### 06.3 文章配图 Skill

写文章要配图，是最烦的重复劳动之一。

这个技能把"读文章 → 选配图类型 → 生成/美化 → 输出"的流程固化。references/ 里放版式骨架和配色规范，scripts/ 里放截图美化、尺寸裁剪的确定性逻辑，AI 只负责判断"这张图配这段文合不合适"。

关键是把"审美判断"写成可执行的约束：不用纯白纯黑、文字不能压脸、AI 生图只在没合适真实图时才用。这些规则看着限制自由，实际是在保护输出下限。

| 技能 | 解决什么痛点 | 标叔的结论 |
|------|------------|-----------|
| 会议纪要 | 每次重新解释格式 | 最该先装 |
| 写作风格 | 文风不统一 | 个人资产首选 |
| 文章配图 | 配图流程重复 | 内容创作者必用 |

三个会抄了？下一章，把它们串成一条线。

## §07 组合用：搭一条多技能工作流

### 07.1 单点能力 vs 系统能力

单个 Skill 解决一个点。多条串起来，才叫系统能力。

举个我自己的例子。写一篇公众号，过去我要：想选题 → 查资料 → 写稿 → 配图 → 做成社交卡片 → 排版。每一步都是一次从头解释。

现在我用一条工作流串起来：选题调研 Skill 查最新资料 → 写作风格 Skill 管文风 → 文章配图 Skill 出图 → 社交卡片 Skill 把段落转成小红书卡片。一条线从头到尾，我只在关键节点点头。

这就是把单点能力变成系统能力。

### 07.2 一条写作流水线的样子

```text
输入：一个选题
  ↓ 选题调研 Skill（查资料、核实事实）
  ↓ 写作风格 Skill（按我的文风起草）
  ↓ 文章配图 Skill（生成/美化配图）
  ↓ 社交卡片 Skill（转成多平台卡片）
输出：一篇稿 + 一套分发素材
```

每一步都是一个独立 Skill，按需加载、用完卸载。主程序始终是轻的，变的是挂在上面的技能。

### 07.3 MCP 接世界，Skill 教方法

这里必须讲清 MCP 和 Skill 的关系。这是 2026 年 Agent 架构里最重要的一对组合，也是 §04.8 那张"通用插座"的落地。

Anthropic 的观察很形象：MCP 像插座，让 Agent 接上外部系统；Skill 像操作手册，让 Agent 知道接上之后怎么干活。

一个 Agent 接入了客户系统、文档系统、日历、财务系统——这只是第一步，它能访问。但它不知道公司内部怎么写报告、客户分级规则是什么、审批怎么走。这些"怎么干"的知识，就是 Skill 的位置。

| 角色 | 负责什么 | 标叔的结论 |
|------|----------|-----------|
| MCP | 连接外部世界（插座） | 让 Agent 够得着工具 |
| Skill | 提供专业知识（手册） | 让 Agent 知道怎么用 |
| Agent | 思考与执行 | 两者都装上才完整 |

MCP 由 Anthropic 在 2024 年 11 月开源，到 2026 年已是 AI 连接外部工具的事实标准。2026 年的 MCP 路线图重点在传输层可扩展、Agent 间通信、治理成熟度、企业就绪。OpenAI 也推出了 Secure MCP Tunnel。标准定了，Skill 才能放心地四处流动。

工作流会搭了。下一章，我们把它工程化。

## §08 工程化：从需求到上线的全流程

### 08.1 Thin Harness, Fat Skills：薄驾驭，重技能

2026 年 Agent 架构有个共识叫"Thin Harness, Fat Skills"——薄驾驭，重技能。

Harness（驾驭层）是跑 Agent 的外层程序，也就是 §04.7 说的"框架"，负责模型循环、文件读写、上下文管理和安全边界。它的核心原则是：别承载复杂业务逻辑，保持极轻。

Skill 反过来，要越做越厚：流程、判断、领域知识、模板、脚本、失败经验，全装进去。

模型只留在它最适合的地方——理解、判断、综合、取舍、表达。确定性的活，下沉给脚本、CLI 或 API。这正是 §04 讲的"框架跑脚本、模型做判断"的分工在工程上的体现。

> **标叔的经验**：这个原则救过我的项目。
>
> 早先我把所有逻辑写进主程序，加一个功能就爆一次上下文。改成"主程序只调度、能力全外挂"之后，系统轻了，维护成本降了一大半。

### 08.2 中心短，辐射厚：信息架构原则

好 Skill 的信息架构，应该是"中心短，辐射厚"。

SKILL.md 只放高信号流程和判断。references/ 放重文档，按条件读取。scripts/ 放确定性逻辑，让 Agent 调用而不是重写。assets/ 放模板、schema、示例。

为什么？因为每个 Skill 都是一种"税"。它进索引后，每个会话、每个用户都在为它的 name 和 description 付上下文成本；被加载后，后续对话都在为主体内容付成本。所以每一句都要问：没有这句，Agent 会不会做错？不会，就删。

### 08.3 eval 与 gotchas：像管代码一样管技能

好 Skill 不是一次写完。它要像代码质量一样维护。

可靠的生命周期是：

1. 先用无 Skill 的 Agent 跑真实任务，找到它会错在哪。
2. 基于真实 query 写 eval（测试集），含正例、反例和"禁止加载"用例。
3. 先调 description，确保该加载时加载、不该加载时不加载。
4. 写主体时删掉显而易见的内容，只留会改变模型行为的判断。
5. 把失败案例追加到 gotchas，而不是不断加长主流程。

Gotchas 是"别这么做"的清单，往往比正向说明更能提升稳定性。比如设计类 Skill 里写："连续三页相同节奏是 P0 错误""文字不能压脸""AI 图只在无合适真实图时使用"。

正向原则模型已经知道，负面边界才是专家经验。

### 08.4 一个完整 Skill 生命周期

把前面收束成一条路径：

```text
1. 发现真实需求（从重复工作开始）
2. 先做一次高质量产物（别先抽象）
3. 抽象流程（识别可复用步骤、输入输出、约束）
4. 工程化模板（固化审美、调用、验证、修复）
5. 跨模型测试（好模型看上限，差模型保下限）
6. 封装发布（GitHub 托管，配 README 和示例）
7. 内容分发（社交平台展示真实结果）
8. 收集反馈（从 issue、评论、数据里找真问题）
```

这条路看着长，本质很简单：每一次真实任务，都不只是在完成任务，而是在积累下一次能调用的能力资产。

工程化懂了。下一章，我们真刀真枪做一个。

## §09 上实战：从零迭代一个数据分析技能

### 09.1 需求：财报异常自动标注

我给自己的任务很具体：上传一份财报，自动提取核心指标、标注异常数据、生成经营分析摘要。

这需求边界清楚，能验证对错，适合做第一个实战技能。

### 09.2 第一版：先用 Agent 跑通

我没先写 Skill，而是先让 Agent 裸跑一次。

我丢了份财报进去，说："提取核心指标，标出异常，写摘要。"它跑出来了，但有三次明显的错：把同比算成了环比、漏标了一个关键异常、摘要里用了我禁止的空话。

这一步的目的不是产出，是找错。无 Skill 的 Agent 会错在哪，就是我要写进 Skill 的内容。

### 09.3 抽象流程，工程化模板

我把流程抽出来：读配置（指标口径）→ 跑分析脚本 → 拿结构化结果 → 按规则标异常 → 按模板写摘要。

稳定动作脚本化：指标计算、同比环比判断、异常阈值比对，全进 scripts/。references/ 放指标口径说明和输出模板。SKILL.md 只写触发、顺序和边界——特别写明"异常必须用红字标在对应指标后"。

gotchas 里我加了三条：不许把同比当环比、环比为负要标"下滑"、摘要禁用"值得一提的是"这类空话。

### 09.4 跨模型测试与迭代

第一版在强模型上跑得不错。换了个弱模型，问题来了：它开始自由发挥格式。

我把输出契约写得更死——references/ 里的 output-format.md 规定了每个字段的位置和类型。弱模型照着填，下限保住了。

> **标叔的经验**：跨模型测试是很多人跳过的一步。
>
> 你以为写好了，是因为你只用最强模型测。真实环境里，调用方用的模型天差地别。好模型看上限，差模型保下限，两个都得测。

做完一个了。下一章，介绍蒸馏 Skill：把书、人、思想都整锅装进 Agent。

## §10 蒸馏 Skill：把书、人、思想都装进 Agent

前几章我们造的是一个一个能用的 Skill。但 2026 年最让我兴奋的变化是另一类工具——它们不做"单个技能"，而是把"原料"整锅**蒸馏**成一个 Skill 包。

一本书、一个人的数字痕迹、一位名人的公开表达，进去，出来的是一个可安装、可版本化、可回滚的 AgentSkills 标准目录。我把它叫"蒸馏 Skill"（distillation skill）。

这条线上现在有五个值得记住的仓库：人→Skill 的 `dot-skill`，书→Skill 的 `book-to-skill`，书/视频/播客→方法论 Skill 集的 `cangjie-skill`，名人/主题→Skill 的 `nuwa-skill`，以及逆向"写书"这件事的 `huashu-bookwriter`。

### 10.1 dot-skill：把一个人蒸馏成 Skill

仓库：`titanwings/colleague-skill`。

它的核心洞察是换了一个问题定义：不去"复现这个人"，而是把选定的"人类痕迹"转化为一个有明确边界、可编辑的技术工件（artifact）。论文把它形式化成 `S = (A, M, L)`：A 是生成的文件，M 是安装元数据，L 是生命周期状态（版本、纠正次数、回滚历史）。

整套系统最狠的设计是**双轨蒸馏**——把 Skill 拆成两条互不干扰又协同工作的轨道：

| 轨道 | 文件 | 装的是什么 | 解决什么问题 |
|------|------|----------|------------|
| **能力轨** | work.md | 工作方法、心智模型、决策启发式、技术标准 | 让它「能干实事」 |
| **人格轨** | persona.md | 沟通风格、互动规则、边界约束、修正记录 | 让它「像这个人」 |

为什么不混一起？因为 CR 时的严格标准是**能力**，怼人时的口头禅是**人格**，混在一起既没法单独调用，也没法单独修正。拆开后系统提供三个独立入口：`/{char}-work`、`/{char}-persona`、`/{char}`，这就是论文说的"可组合性"。

它还支持三大家族预设：`colleague`（同事，6 层人格 + Work 模块，原料优先长文写作）/`relationship`（亲密关系，情感模式四件套：表达 DNA、情绪触发点、冲突模式、修复模式）/`celebrity`（名人，六维度研究档案）。

做出来之后还能进化：增量合并（追加新文件不覆盖已有结论）、对话纠正（你说"他不会这样"，立刻写入 Correction 层生效）、版本回滚（每次更新自动归档）。

### 10.2 book-to-skill：把一本书编译成 Skill

仓库：`virgiliojr94/book-to-skill`。

它最反直觉的地方是把"检索"这件事的时机从**查询期**挪到了**编译期**。传统 RAG 是"你问→切分→embedding→找相似块→塞进 prompt"，本质还是"在原文里捞针"。book-to-skill 是提前把作者花了几年搭的框架、原则、心智模型、反模式一次性抽出来。所以它管自己叫 **compile-time knowledge compiler**——编译一次，永久复用。

它解决的是一个被忽略的成本，作者叫它 **Discovery Loop Tax（探索循环税）**。一本 256K token 的技术书，传统翻阅单次问答要 77,866 token；编译后只要 ~5000 token（4000 核心 + 1000 章节），差 15 倍以上。

代码库刻意保持极简，边界切得干净：

| 组件 | 谁干 | 干什么 |
|------|------|--------|
| extract.py | Python 确定性代码 | 格式路由、编码检测、文本提取、元数据 |
| SKILL.md | AI Agent（Claude/Amp） | 结构分析、框架提取、摘要、生成 Skill 文件 |

两者只通过临时目录里的 `full_text.txt` + `metadata.json` 两个文件通信。Python 不依赖 AI API，AI 也不用操心 PDF 怎么解析。

生成的 Skill 包用 **progressive disclosure**（渐进式揭露）压 token：平时只载 `SKILL.md`（~4000 token），你问 `ch05` 才载 `chapters/ch05`（~1000 token），问术语才翻 glossary，要决策规则才看 cheatsheet。一本 400 页的书编译一次约 $1（Claude Sonnet 4.5）。

### 10.3 nuwa-skill：把名人的"思维底座"装进 Agent

仓库：`alchaincyf/nuwa-skill`，作者花叔。

作者自己给的定位很犀利——"`colleague-skill` 蒸馏人做什么，女娲蒸馏人怎么想"。所以它不是角色扮演，而是把乔布斯、芒格、马斯克、费曼这些人的**认知操作系统**抽成可运行 Skill。

普通角色扮演只抄语气皮毛，女娲要的是思维底座。两者差异：

| 层级 | colleague-skill | nuwa-skill |
|------|----------------|-----------|
| 做什么（工作习惯、流程） | ✅ | — |
| 怎么说话（语气、口癖） | ✅ | ✅ 表达 DNA |
| **怎么想**（心智模型、认知框架） | — | ✅ |
| **怎么判断**（决策启发式） | — | ✅ |
| **什么不做**（反模式、价值观底线） | — | ✅ |
| **知道局限**（诚实边界） | — | ✅ |

一个观点想升级成"心智模型"，必须同时过三关：

1. **跨域复现**：在 2+ 个不同领域都出现过，不是随口一说。
2. **有预测力**：能推断此人对新问题的立场，不是死记硬背。
3. **有排他性**：不是所有聪明人都会讲的套话，要有辨识度。

三关全过 → 升格为心智模型；过 1-2 关 → 降级为决策启发式；一关不过 → 直接丢弃。这套过滤保证出来的不是成功学鸡汤，而是这个人真正独特的思维骨架。

出厂前还要过两道测试：3 个公开问题（此人公开回答过的问题，方向必须一致）+ 1 个未讨论问题（他没说过的话题，Skill 应表现"适度不确定"而非斩钉截铁）。第二道专门防过拟合——对没说过的事也斩钉截铁，说明它背的是语料，不是思维框架。

它还内置了**达尔文闭环**：8 维度评估 + 棘轮机制（分数只能升不能降）+ 独立评分（改 Skill 的 Agent 不能自己打分）+ 人在回路。改完变差就 `git revert`，反哺回来的共性问题会再喂给生成流程。

### 10.5 huashu-bookwriter：逆向"写书"这件事

仓库：`ZeroxZhang/huashu-bookwriter`。

前四个都是"把某个对象变成 Skill"，huashu-bookwriter 是倒过来的——它从一个人的书和仓库里**逆推出"他怎么写书"**，再把这个方法论装回去生成新书。ZeroxZhang 逆向花叔的 4 本橙皮书 + 开源仓库，还原度 95%+。

它不是写作 Prompt，是**写作操作系统**——管的是从选题→大纲→调研→撰写→QC→PDF 导出的全生命周期。逆向出来的核心是 5 层精密结构：

| 层级 | 核心规则 | 解决什么问题 |
|------|----------|------------|
| **渐进式信任建立** | 每章前 2-3 段必须有时间线锚点；没体验就诚实标注 | 让读者相信你真做过 |
| **结构化知识路径** | Part1 从零跑通 → Part2 核心能力 → Part3 多场景实战 | 避免「读完还是不会」 |
| **风格 DNA** | 单句 ≤25 字、第一人称高频、具体数字、禁用词表 | 让不同 Agent 写出来像同一个人 |
| **TDD 式质量铁律** | 每章 12 项 QC + 全书 10 项 QC，不通过就重写 | 自动化关卡，违反即删除重写 |
| **Agent Protocol** | 写作前必须先调研确认事实，不准凭训练数据编造 | 治 AI 幻觉 |

第三层"风格 DNA"最狠——它不光规定"该用什么"，更规定"绝对不能用什么"：「接下来我们将进行」→「我们来」、「进行操作」→「点击」、「综上所述」「值得注意的是」「强大的」「革命性的」全在禁用词表里。禁用词的本质不是词不好，而是这个词意味着你在逃避某种表达责任。

> **标叔的结论**：风格一致性不是靠天赋，是靠规则。
>
> 这五类 Skill 把"人怎么想、书怎么写、流程怎么跑"全蒸馏成了可分发工件，本质上它们在做同一件事——把软资产硬化成硬组件。

### 10.6 怎么串起来用

五条线不是互斥的，可以并存、可以组合。比如让：

- `nuwa-skill/芒格` 管投资判断的认知视角
- `cangjie-skill/穷查理宝典` 管芒格同款思维方法的可调用 Skill 集
- `book-to-skill/DDIA` 管技术框架的知识体系
- `colleague-skill/你同事` 管 Code Review 的表达口味
- `huashu-bookwriter` 管从大纲到 PDF 的写作流程

同场协作，一个负责"怎么想"、一个负责"用什么框架"、一个负责"怎么说"、一个负责"怎么交付"——那就是真正意义上的"智囊团 + 写作流水线"了。

### 10.7 三个必须知道的边界

这些工具的作者自己反复强调几个"不"，我抄过来给你：

- **不复制人**。nuwa 提的是认知框架，不是把这个人"复活"；colleague 提的是有用的判断力，不是数字永生。
- **不 ship 任何书的内容**。book-to-skill / cangjie-skill 只生成结构化衍生笔记（框架名、定义、takeaway），不复制原文。
- **不解决海量跨库检索**。50+ 本书跨库是 RAG / NotebookLM 的地盘；单本深挖、反复用才是 Skill 的地盘。

版权要自己扛——第三方书生成的 Skill 别 redistribution，内部文档、自有写作、开源许可材料没问题。信源决定上限——一手材料（本人书/长访谈）远胜二手解读，批评视角必须保留。

你给的是"原材料"，它给回你的是一个可安装、可版本化、可回滚的 AgentSkills 标准目录。这就是蒸馏 Skill 的本质。

## §11 结语：从菜谱到厨艺

### 11.1 模型是处理器，Runtime 是系统，Skill 是应用

Anthropic 用一个计算机史的类比收尾，我觉得讲得最透：

模型像处理器。它需要巨大投入，也有巨大潜力，但处理器本身解决不了所有问题。后来人类做了操作系统，让处理器更有价值——它能调度进程、资源和数据。

在 AI 里，Agent Runtime（运行环境，也就是本书说的"框架 / harness"）正在扮演操作系统的角色。它负责把正确的 token 送进模型，把模型输出组织成能执行的动作。

但有了平台之后，真正的价值来自应用。处理器和操作系统只由少数公司构建，应用却由大量开发者做出。这些应用里，装着不同领域的专业知识和经验。

Anthropic 希望 Skills 打开的，就是这个应用层。让更多人把自己的专业能力、流程经验、解决问题的方法，放进一个文件夹，交给 Agent 使用。

这就是为什么他们说："不要继续重做 Agent，开始做 Skills。"

### 11.2 最稀缺的，是可复用的能力组织方式

Agent 时代最稀缺的，不是更聪明的模型，也不是更炫的界面。

是能把人的经验、工作流和品味，变成一种可以分发、调用、评价和持续迭代的商品。Skill 第一次让这件事变得可能。

它不只能给技术团队用。财务、招聘、法律、健身、餐饮、活动策划——每个有专业流程的人，都能把自己的"怎么干"变成可调用的能力。

> **标叔的经验**：这半年我最大的体会就一句。
>
> 真正拉开差距的，不是你会不会用某个工具，而是你有没有把自己的经验，沉淀成别人（和 Agent）能复用的资产。

### 11.3 你该从哪里开始

别想太宏大了。

今晚就做一件小事：挑一个你每周都重复、又懒得每次解释的任务。按 §05 的五步，把它拆成一个 Skill。会议纪要、写作风格、文章配图——这三个随便选一个，先跑通。

跑通一个，你就懂了八成。剩下的，是在这个基础上长出来的。

软件时代，我们下载 App。Agent 时代，我们加载 Skills。先上手、先积累 Skill 的人，会比别人早一步建立真正的护城河。

去造你的第一个吧。
