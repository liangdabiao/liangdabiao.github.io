# 全面调研，这个库的 原理逻辑 具体是什么：： https://github.com/titanwings/colleague-skill

先把结论摆在前面：**dot-skill 不是「把人做成 AI 分身」，而是把一个人在各类数字痕迹里的「可复用判断力」蒸馏成一个可检查、可修正、可安装、可回滚的 Skill 包**。它真正的创新不在 Prompt 拟人，而在把「人物知识」当成一种**有边界的技术工件（artifact）**来处理——这是理解整套原理的钥匙。

下面按「总逻辑 → 流水线 → 三大家族 → 双轨内部结构 → 进化机制 → 部署与局限」六层拆开。

## 一、总逻辑：从「记忆存储」到「痕迹→Skill 工件」

传统做法有两种，都不到位：

- **记忆系统**：存了一堆聊天片段，但没法结构化复用
- **人设系统**：把知识、判断、语气混在一起，最后变成「似是而非的模仿」

dot-skill 的核心洞察是换了一个问题定义：不去「复现这个人」，而是把**选定的人类痕迹转化为一个有明确边界、可编辑的技术产物**。论文把它形式化成一个 artifact 问题——给定用户描述 p、源范围 c、源材料 D，系统输出一个 Skill 包 S = (A, M, L)：

- **A**：生成的文件（SKILL.md、work.md、persona.md 等）
- **M**：机器可读的元数据和安装信息
- **L**：生命周期状态（版本、纠正次数、回滚历史）

这个包必须满足五个操作属性：**可移植、可检查、可组合、可纠正、可治理**。这五个词基本就是整套系统的设计宪法。

## 二、主流水线：四阶段「蒸馏」全过程

整个系统是一条明确的 pipeline，从原材料到可调用 Skill 要走完四步：

### 1️⃣ 痕迹摄取（Trace Intake）—— 把异构数据规范化

先把散落在各处的原材料归一化。支持的数据源覆盖了三类家族的不同需求：

| 数据源 | 消息 | 文档/Wiki | 适用场景 |
|---|---|---|---|
| 飞书 | API 自动 | ✅ | colleague，输入名字全自动 |
| 钉钉 | 浏览器 | ✅ | colleague，API 不支持历史消息 |
| Slack | API | — | colleague，需管理员装 Bot |
| 微信 | SQLite 导出 | — | relationship |
| PDF/图片/邮件/Markdown | 手动上传 | ✅ | 通用 |

这一步的关键是**保留源边界**——每条证据后面都要能追溯到原始材料，而不是把东西一股脑塞进 prompt。

### 2️⃣ Preset 路由 —— 选三大家族之一

用户启动 `/dot-skill` 后，先选本次蒸馏的对象类型：`colleague` / `relationship` / `celebrity`。这个选择会落到 `tools/skill_presets.py` 的注册表里，决定后面走哪条 prompt 管道、哪种采集策略、哪种生成模板。**同一条流水线，三套配置**。

- colleague：intake → work_analyzer → persona_analyzer → work_builder → persona_builder
- relationship：情感/冲突/修复专用 prompt
- celebrity：六维度研究工具链（字幕下载 → 转录清洗 → 研究合并 → 质量检查）

### 3️⃣ 双轨蒸馏（Dual-track Distillation）—— 最核心的设计

这是整套系统最狠的地方：**把 Skill 拆成两条互不干扰又协同工作的轨道**，避免传统人设系统「知识+语气+判断混成一锅粥」。

| 轨道 | 文件 | 装的是什么 | 解决什么问题 |
|---|---|---|---|
| **能力轨 Capability Track** | work.md | 工作方法、心智模型、决策启发式、技术标准 | 让它「能干实事」 |
| **人格轨 Behavior Track** | persona.md | 沟通风格、互动规则、边界约束、修正记录 | 让它「像这个人」 |

为什么要拆？因为不拆就会乱——CR 时的严格标准是**能力**，怼人时的口头禅是**人格**，混在一起既没法单独调用，也没法单独修正。拆开后系统提供三个独立入口：`/{char}-work`（只要能力）、`/{char}-persona`（只要人格）、`/{char}`（全开），这就是论文说的**可组合性**。

### 4️⃣ 工件生成与产品化 —— 变成标准 Skill 包

最后由生成器把双轨渲染成符合 **AgentSkills 开放标准**的目录结构：

```text
生成的 Skill/
├── SKILL.md          # 官方入口（frontmatter + 双轨整合）
├── work.md           # 能力轨
├── persona.md        # 人格轨（含 6 层结构 + Correction）
├── manifest.json     # 安装元数据
├── meta.json         # 版本/来源/治理信息
└── 可独立调用的子技能
```

这个包可以直接安装到 Claude Code / Hermes / OpenClaw / Codex 任意宿主，也可以发布到社区 gallery。

## 三、三大家族的「人格底座 + 附加模块」差异

dot-skill 用 **Persona 做通用底座，不同家族挂不同附加模块**，运行逻辑统一为：接到任务 → Persona 判断态度与语气 → 附加模块补齐执行细节 → 用他的方式输出。

### 🧑💼 colleague：双层架构 + 6 层人格

Persona 拆成 6 层（硬规则 → 身份 → 表达风格 → 决策模式 → 人际行为 → Correction），再挂 Work Skill 模块（负责范围、工作流程、输出偏好、经验知识库）。原材料优先级：**长文写作（设计文档/CR 评论）> 决策回复 > 群聊碎话**。

### 💞 relationship：情感模式四件套

不挂 Work 模块，Persona 里重点抓表达 DNA、情绪触发点、冲突模式、修复模式。Roadmap 还规划了照片分享、「今天我过得怎样」的生活化回复。**时间跨度越长越好，最好覆盖冲突+修复两种场景**。

### 🌟 celebrity：六维度研究档案

最重的一家，Persona + 心智模型/决策启发式/表达 DNA/外部评价对照，再挂六维度研究档案（著作/访谈/决策/时间线等）。工具链是 `download_subtitles.sh → srt_to_transcript.py → merge_research.py → quality_check.py`，**优先一手材料（本人书/博客/长访谈），少喂二手解读**。

## 四、生成之后还能「进化」：三条机制

Skill 包不是一次生成就完事，它有一套生命周期管理，这是它区别于普通人设机器人的关键。

1. **增量合并（merger）**：追加新文件 → 自动分析 delta → 合并进对应章节，**绝不覆盖已有结论**。由 `prompts/merger.md` 和 `tools/version_manager.py` 驱动。
2. **对话纠正（correction）**：你对它说「他不会这样，他应该是 xxx」→ 写入 Correction 层，**立即生效**。由 `correction_handler.md` 解析自然语言反馈，生成 Markdown 补丁。
3. **版本控制**：每次更新自动归档，支持回滚到任意历史版本；quality_check.py 还会检查心智模型覆盖率、风格一致性、内部矛盾、溯源链接和版权合规。

## 五、部署：多宿主兼容的「元 Skill」

dot-skill 本身是一个**元 Skill（meta-skill）**——它的作用是生成别的 Skill。安装时让 Agent 自己克隆到对应目录：

- Claude Code → `~/.claude/skills/dot-skill`
- OpenClaw → `~/.openclaw/workspace/skills/dot-skill`
- Codex → `~/.codex/skills/dot-skill`
- Hermes → 克隆后跑 `install_hermes_skill.py`

生成的具体角色 Skill 也能一键装回任意宿主。换句话说，**dot-skill 是工厂，生成的 colleague/relationship/celebrity 才是产品**。

## 六、原理边界：它到底不是什么

这一点论文反复强调，也是很多人会误读的地方：

> **它不是「数字永生」或「复刻一个人」**，而是把「有用的判断力」打包成可治理的技术工件。所以它刻意保留源边界、修正记录、回滚历史和治理元数据——这些东西对一个「复刻人」没用，但对一个「可检查的 Skill 包」至关重要。

所以整个库的原理逻辑可以压缩成一句话：**用 Preset 路由把异构痕迹导进三条家族管道 → 用双轨蒸馏把能力和人格分开提取 → 用标准 Skill 包封装成可组合、可纠正、可回滚的工件 → 通过多宿主部署变成真正可调用的 Agent 能力**。

要不要我接着帮你拆一下「celebrity 六维度研究工具链」每一步具体怎么跑，或者直接拿 Karpathy 那个 demo 走一遍完整流水线？


# 继续 调研  book-to-skill

先把一个容易踩坑的点说在前面：你给的 `titanwings/colleague-skill` 是上一轮那个「人物痕迹蒸馏」库，但 **book-to-skill 的实际仓库是 `virgiliojr94/book-to-skill`**，两者都瞄准 AgentSkills 标准，但一个是「人→Skill」、一个是「书→Skill」，是同一套哲学下的两条产品线。下面接着上一轮的框架往下拆。

## 一句话定位：它是「编译期知识编译器」，不是 RAG

book-to-skill 最反直觉的地方在于**把「检索」这件事的时机从查询期挪到了编译期**。

传统 RAG 是「你问 → 切分 → embedding → 找相似块 → 塞进 prompt」，本质还是「在原文里捞针」；book-to-skill 是**提前把作者花了几年搭的框架、原则、心智模型、反模式一次性抽出来，变成结构化的 Skill 包**。前者回答「哪里提到了 X」，后者回答「作者关于 X 的 12 个框架，拿来推理」。

所以它管自己叫 **compile-time knowledge compiler**——编译一次，永久复用。

## 一、总逻辑：为什么是「编译期」而不是「查询期」

这里藏着一个被忽略的成本，作者叫它 **Discovery Loop Tax（探索循环税）**。

如果一个 Agent 每次都直接读 PDF 回答问题，它的行为是：读目录 → 跳章节 → 遇到不懂的术语 → 往回翻 → 再往前找 → 每一轮跳转都堆进上下文。实测一本 256K token 的技术书，传统翻阅单次问答要 **77,866 token**；而 book-to-skill 编译后只要 **~5000 token**（4000 核心 + 1000 章节），差 15 倍以上。即便一次性探索，也比反复翻 PDF 省 2.4–15.6 倍。

> 它的核心洞察：**把「找到正确章节」的成本从执行期预付到编译期，执行期只做「按图索骥」**。

## 二、双引擎架构：SKILL.md + extract.py 各管一段

整个代码库刻意保持极简，只有两个核心部分，边界切得很干净：

```text
book-to-skill/
├── SKILL.md              # 声明式编排层（522行）：10步流水线，给AI Agent读的指令
├── scripts/
│   ├── extract.py        # 命令式提取引擎（819行）：确定性格式解析
│   └── extractor/        # 模块化解析包
│       ├── parsers/      # pdf/epub/docx/html/rtf/calibre/text
│       ├── dependencies.py
│       └── utils.py
├── tests/                # pytest 套件
├── README.md
└── LICENSE.md             # MIT
```

**为什么非要拆？** 因为两件事需要的「脑子」完全不同：

| 组件 | 谁干 | 干什么 | 为什么给它 |
|---|---|---|---|
| extract.py | Python 确定性代码 | 格式路由、编码检测、文本提取、元数据 | 需要精确、可测试、可审计 |
| SKILL.md | AI Agent（Claude/Amp） | 结构分析、框架提取、摘要、生成 Skill 文件 | 需要自然语言理解和推理 |

两者**只通过临时目录里的两个文件通信**：`full_text.txt` + `metadata.json`。这意味着 Python 代码完全不依赖 AI API，AI 也永远不用操心 PDF 内部怎么解析——边界极其干净。

## 三、10 步流水线：从 PDF 到 Skill 的完整路径

SKILL.md 里硬编码了一条 10 步流水线，Agent 会严格照着走：

1. **接收输入**：文件 / 文件夹 / glob / 多文件混合，支持 PDF、EPUB、DOCX、TXT、MD、rst、AsciiDoc、HTML、RTF、MOBI/AZW 等 11 种格式
2. **类型判断**：技术类（代码/表格/公式）vs 文字类（散文为主）——这步决定用哪个解析器
3. **调用 extract.py**：技术书走 **Docling**（~1.5 秒/页，保留代码块和表格）；文字书走 **pdftotext/pypdf**（秒级）
4. **输出 full_text.txt + metadata.json**：含页数、字数、预估 token、章节检测结果、目录是否存在
5. **确认内容类型与成本预估**：Agent 问你是技术书还是文本书，给 discovery tax 对比
6. **结构分析**：提取书名、作者、章节、目录，识别作者的核心框架
7. **逐章摘要**：每章 800–1200 tokens，技术书额外保留代码示例和参考表
8. **生成三大参考文件**：术语表、模式清单、速查表
9. **合成主 SKILL.md**：核心心智模型 + 章节索引，约 4000 tokens
10. **写入 Skill 目录并清理临时文件**：部署到 `~/.claude/skills/` / `~/.copilot/skills/` / `~/.agents/skills/`

## 四、输出结构：渐进式揭露（Progressive Disclosure）

这是它能把 token 压到 1/15 的关键设计。生成的 Skill 包长这样：

```text
your-book-skill/
├── SKILL.md          # 核心心智模型 + 章节索引，~4000 token，常驻 context
├── chapters/
│   ├── ch01-*.md     # 每章摘要，~1000 token/章，命中才加载
│   └── ch05-*.md
├── glossary.md       # 术语表（字母序 + 章节引用），~1500 token，查词才加载
├── patterns.md       # 技术/算法/设计模式汇总，~2000 token，按需
├── cheatsheet.md     # 决策表 + 速查规则，~1000 token，按需
└── metadata.json     # 书籍元数据
```

加载逻辑是**分层揭露**：

- 平时只载 SKILL.md（~4000 token）→ Agent 知道「这本书讲什么、有哪些框架、每章在哪」
- 你问 `ch05` → 只载 chapters/ch05（~1000 token）
- 你问某个术语 → 只载 glossary 对应段
- 你要决策规则 → 只载 cheatsheet

这就是所谓 **progressive disclosure**：先给索引，按需展开。对比一下三种方案的代价：

| 方案 | 时机 | Token | 理解能力 | 幻觉 | 适用 |
|---|---|---|---|---|---|
| 直接灌 PDF | 每次加载全文 | 20 万+ | 差，长文本丢信息 | 高 | 临时翻阅 |
| 传统 RAG | 查询时切分检索 | 中等 | 片段匹配，无整体框架 | 中 | 海量跨书检索 |
| book-to-skill | 编译期一次性结构化 | ~5000 | 完整知识体系 | 极低 | 单本/成套深度复用 |

## 五、它和 dot-skill / RAG 的根本差异

接上一轮聊的 `colleague-skill`，这三者的关系要分清：

**book-to-skill vs RAG**
- RAG 优化的是「帮我找到提到 X 的地方」，book-to-skill 优化的是「用书里的框架来推理」
- 50+ 本书跨库检索 → RAG 赢；单本深挖、工作流里反复用 → Skill 赢

**book-to-skill vs dot-skill（你上一轮调研的那个）**
- dot-skill 是「人→Skill」：从聊天记录/数字痕迹蒸馏人的判断力和人格，偏「人物建模」
- book-to-skill 是「书→Skill」：从书籍/文档提取作者的框架和知识体系，偏「知识编译」
- 两者都输出 AgentSkills 标准目录，可以共存——比如用 dot-skill 建一个「我的导师」Skill，用 book-to-skill 建一个「DDIA」Skill，Agent 同时调用

**book-to-skill vs 直接靠训练数据**
- 热门书（Clean Code、DDIA）Claude 确实「知道」，但是互联网平均过的压缩版，会 hallucinate 具体引语和章节位置
- book-to-skill 从你自己的副本编译，每个框架名、反模式列表、章节号都 grounding 到你提供的文本
- 对小众技术书、公司内部文档、新出版物、翻译作品——这些是 Claude 训练数据里没有的，它优势更大 

## 六、使用边界：它自己反复强调的几个「不」

这部分很重要，因为很容易被营销话术带偏：

- **不 ship 任何书的内容**：它只生成结构化衍生笔记（框架名、定义、takeaway），不复制原文，Quality Rule #7 明确禁止拷贝原始段落
- **不替你读书**：生成的 Skill 是「你的笔记」，不是书的替代品；Book2Skills 那个同类项目甚至要求贡献者必须读完原书，二手摘要会被拒 
- **不解决海量跨库检索**：那是 RAG / NotebookLM 的地盘，它定位是「深耕一本、融入工作流」
- **版权要自己扛**：第三方书生成的 Skill 别 redistribution，内部文档/自有写作/开源许可材料没问题
- **本地处理**：extract.py 全本地跑，文件不会被工具上传；但后续 Agent 把文本喂给云端模型时，仍遵守对应提供商的数据条款

## 七、典型用法

```bash
# 安装（两种别搞混）
# 要 /book-to-skill 斜杠命令 → git clone 进 skills 目录
git clone https://github.com/virgiliojr94/book-to-skill.git ~/.claude/skills/book-to-skill
# 只要 CLI 提取器 → pip install（不会注册 agent skill）
pip install "book-to-skill[pdf,epub,docx]"

# 单本书
/book-to-skill ~/books/ddia.pdf

# 多文件合成一个 Skill
/book-to-skill ~/papers/paper1.pdf ~/notes/export.txt unified-research

# 整个目录
/book-to-skill ~/workspace/project-docs/ project-knowledge

# glob
/book-to-skill "~/books/*.epub" my-library

# 增量更新已有 Skill
/book-to-skill ~/articles/new-paper.pdf ~/.claude/skills/project-knowledge

# 运行时调用
/ddia                      # 加载核心心智模型
/ddia ch05                 # 直接读第五章
/ddia replication          # 检索 replication 主题
```

成本方面，一本 400 页的书编译一次约 **$1**（Claude Sonnet 4.5），比每次对话都灌 PDF 便宜太多。

所以 book-to-skill 的原理可以压缩成一句话：**用 extract.py 做确定性格式解析、用 SKILL.md 流水线做 AI 结构化推理，把「书的框架」在编译期抽成 AgentSkills 标准目录，运行时靠 progressive disclosure 只加载命中的那一块，从而把 discovery tax 从每轮问答压到一次性预付**。

接着上一轮的 `colleague-skill`（人→Skill）和 `book-to-skill`（书→Skill），`nuwa-skill` 是这个家族里**第三条产品线：名人/主题→Skill**。作者花叔自己给的定位很犀利——「同事.skill 蒸馏人做什么，女娲蒸馏人怎么想」。所以它不是角色扮演，而是把乔布斯、芒格、马斯克、费曼这些人的**认知操作系统**抽成可运行 Skill。

下面按「定位差异 → 四步流水线 → 五层提取 → 诚实边界 → 达尔文进化 → 安装与生态」拆开。

## 一、它和 colleague-skill / 角色扮演的根本差异

很多人第一反应是「这不就是让 AI 扮演乔布斯吗」——错。普通角色扮演只抄语气皮毛，nuwa 要的是**思维底座**。作者用一张对比表说清楚了：

| 层级 | colleague-skill | nuwa-skill |
|---|---|---|
| 做什么（工作习惯、流程） | ✅ | — |
| 怎么说话（语气、口癖） | ✅ | ✅ 表达 DNA |
| **怎么想（心智模型、认知框架）** | — | ✅ |
| **怎么判断（决策启发式）** | — | ✅ |
| **什么不做（反模式、价值观底线）** | — | ✅ |
| **知道局限（诚实边界）** | — | ✅ |

一个是「行为录像带」，一个是「认知操作系统」。

## 二、四步蒸馏流水线：从名字到可运行 Skill

输入一个名字，女娲自动走完四步，全程有 6 个 Agent 并行跑：

### 第一步：六路并行采集（Swarm Intelligence）

6 个 Agent 同时从 6 个维度抓公开资料，各自独立存档：

- **著作**：书籍、系统长文、论文 → 抓反复出现的核心论点
- **播客/访谈**：演讲、深度采访、播客 → 抓被追问时的即兴思维
- **社交媒体**：Twitter/X、微博、博客 → 抓语言 DNA 和情绪反应
- **批评者视角**：反对意见、批评文章、传记 → 反向验证，避免粉丝滤镜
- **决策记录**：关键商业动作、转折案例 → 看「言行是否一致」
- **人生时间线**：成长节点、思想转变 → 抓认知演化，不只抓静态结论

关键是第 4 路「批评者视角」——这是它避开普通角色扮演「只会复读金句」的核心设计。**只看好话，蒸馏出来的是信徒；正反一起看，蒸馏出来的才是人**。

### 第二步：三重验证提炼（最狠的过滤器）

一个观点想升级成「心智模型」，必须同时通过三关，缺一不可：

1. **跨域复现**：在 2+ 个不同领域都出现过，不是随口一说
2. **有预测力**：能推断此人对新问题的立场，不是死记硬背
3. **有排他性**：不是所有聪明人都会讲的套话，要有辨识度

三关全过 → 升格为心智模型；过 1-2 关 → 降级为决策启发式；一关不过 → 直接丢弃。**这套过滤保证了出来的不是成功学鸡汤，而是这个人真正独特的思维骨架**。

### 第三步：构建 Skill 文件

验证完组装成标准 `SKILL.md`，固定包含五块：

- **3-7 个心智模型**：核心认知框架，比如乔布斯的「聚焦即说不」「端到端控制」
- **5-10 条决策启发式**：做判断的具体规则
- **表达 DNA**：语气、节奏、用词偏好
- **价值观与反模式**：他坚信什么、绝不做什么
- **诚实边界**：明确标注这个 Skill 做不到什么

### 第四步：质量验证

出厂前还要过两道测试：

- **3 个公开问题测试**：此人公开回答过的问题，方向必须一致
- **1 个未讨论问题测试**：他没说过的话题，Skill 应表现「适度不确定」而非斩钉截铁

第二道专门防**过拟合**——如果一个 Skill 对没说过的事也斩钉截铁，说明它背的是语料，不是思维框架。

## 三、五层提取：深度决定它不是角色扮演

普通角色扮演只有「语气」一层，女娲拆五层，越往后越接近认知内核：

| 层级 | 提取什么 | 解决什么问题 |
|---|---|---|
| 表达 DNA | 语气、节奏、用词偏好 | 像这个人说话 |
| 心智模型 | 核心认知框架 | 像这个人想问题 |
| 决策启发式 | 具体判断规则 | 像这个人下结论 |
| 反模式/价值观 | 什么绝不碰 | 像这个人守底线 |
| 诚实边界 | 明确标注做不到 | 不忽悠用户 |

**一个不告诉你局限在哪的 Skill，不值得信任**——这是作者反复强调的原则。具体边界包括：蒸馏不了直觉和灵感、只能捕捉调研截止前的快照、公开表达≠真实想法、不虚构未公开决策。

## 四、保真度评分卡：15 个官方 Skill 全员 A

这不是营销话术，它有硬性评估体系。官方 15 个 Skill 全部通过**独立双 Agent 盲测**，按立场一致性、风格辨识度、边缘诚实度、来源透明度、结构完整度打分，全员 A 级（≥85 分）：

> MrBeast/纳瓦尔/塔勒布/乔布斯/Karpathy/Paul Graham/张雪峰 97 · 芒格/费曼/X 导师 96 · 特朗普 95 · Ilya 94 · 张一鸣 93 · 孙宇晨 91 · 马斯克 89

每个 Skill 目录内都有 `FIDELITY.md` 可查完整评分卡，方法论在 `references/fidelity-scorecard.md`。

## 五、达尔文闭环：造完还能自己进化

这是 nuwa 区别于同类项目的关键——**女娲负责「造」，达尔文负责「磨」**，而且女娲 Phase 5 双 Agent 精炼本身就内置了达尔文的评估体系。

达尔文的机制受 Karpathy `autoresearch` 启发，把「优化训练代码」改成「优化 SKILL.md」：

1. **8 维度评估**：结构维度 60 分 + 效果维度 40 分，实测表现权重最高 25 分
2. **棘轮机制**：分数只能升不能降，改完变差就 `git revert`
3. **独立评分**：改 Skill 的 Agent 不能自己评分，避免「年终自评打超出预期」
4. **人在回路**：每轮展示 diff + 分数变化，等用户确认再继续
5. **反哺生成**：达尔文批量优化发现的共性问题（边界缺失、frontmatter 太短、引用路径不存在）会反哺回女娲的生成流程

所以 nuwa 生成的每个 Skill **出厂就经过了一轮进化**，不是一次性产物。

## 八、使用边界：它自己反复强调的几个「不」

花叔不公开的写作 Skill，我逆向出来了

接着前三轮的 skill 家族（`colleague-skill` 人→Skill、`book-to-skill` 书→Skill、`nuwa-skill` 名人→Skill），`huashu-bookwriter` 是这条线上**反向的那一环：从「花叔已出版的书 + 开源仓库」逆推出「写书的 Skill」**——相当于把一个人的写作操作系统整个拆下来，装进 Claude Code 里当流水线用。作者 ZeroxZhang 自己叫它「花叔橙皮书创作 Skill（逆向工程版）」，实测橙皮书类还原度 95%+ 。

## 一句话定位：它不是写作 Prompt，是「写作操作系统」

普通写书 Prompt 只管「帮我写一章」，huashu-bookwriter 管的是**从选题→大纲→调研→撰写→QC→PDF 导出的全生命周期**，并且把花叔的风格 DNA、质量铁律、Agent 协作协议全部固化成了可检查的 rules。

它的核心洞察其实和前几个库一脉相承：**把「写作这门手艺」当成一个有边界、可编辑、可复现的技术工件（artifact）来处理**——只不过这次逆向的对象不是「人」「书」「认知」，而是「一个人写书的方法论」。

## 一、五层体系：风格一致性靠规则，不靠天赋

这是整个库最值钱的部分，逆向出了花叔写作的 5 层精密结构 ：

| 层级 | 核心规则 | 解决什么问题 |
|---|---|---|
| **渐进式信任建立** | 每章前 2-3 段必须有时间线锚点；没体验就诚实标注，不编造 | 让读者相信你真做过 |
| **结构化知识路径** | Part1 从零跑通 → Part2 核心能力 → Part3 多场景实战，每章只解决一件事 | 避免「读完还是不会」 |
| **风格 DNA**（最关键） | 单句≤25 字、第一人称高频、具体数字、禁用词表 | 让不同 Agent 写出来的像同一个人 |
| **TDD 式质量铁律** | 每章 12 项 QC + 全书 10 项 QC，不通过就重写；配 Rationalization Table 防 Agent 找借口 | 自动化关卡，违反即删除重写 |
| **Agent Protocol** | 写作前必须先调研确认事实，不准凭训练数据编造 | 治 AI 幻觉 |

第三层「风格 DNA」是最狠的——它不光规定「该用什么」，更规定「**绝对不能用什么**」：「接下来我们将进行」→「我们来」、「进行操作」→「点击」、「综上所述」「值得注意的是」「强大的」「革命性的」全在禁用词表里。禁用词的本质不是词不好，而是**这个词意味着你在逃避某种表达责任** 。

## 二、七步流水线：从「写一本书」到 PDF 的全过程

触发词很简单——「写一本书」「帮我写个指南」「创作 XX 从入门到精通」「做个橙皮书」，Skill 会自动唤醒，走完这 7 步 ：

1. **选书籍类型** → Type A/B/C 三选一
2. **确认选题** → 目标读者、信息密度、适用场景
3. **设计大纲** → Part 结构 + § 编号章节，有 `outline-schema.json` 管数据结构
4. **信息搜集** → 官方文档、社区资源，必须调研确认事实
5. **章节撰写** → 按 3 种章节模板（概念讲解型 / 实战教程型 / 深度分析型）走，遵守风格 DNA
6. **审校 QC** → 12+10 项检查点，不通过就重写
7. **PDF 导出** → 封面、目录、内容、代码块、分页一气呵成

每步都有**明确输入输出和确认关卡**——未经你确认，流程不会偷偷往下走，这是它和「一次性 Prompt」最大的差别 。

## 三、三种书籍蓝图：不同场景走不同结构

| 类型 | 结构 | 适用场景 |
|---|---|---|
| **Type A 从入门到精通** | 10±2 章、3 Part、双语标题 | 系统教程、完整技术手册 |
| **Type B 橙皮书** | 17-35 节、5+ Part、代码密集 | 深度技术文档、架构拆解 |
| **Type C 快速指南** | 12-17 章、概念+实战 | 安装指南、上手手册 |

对应的三本原版就是 Claude Code / OpenClaw / Hermes Agent「从入门到精通」系列——每本都严丝合缝对得上蓝图 。

## 五、PDF 导出链：pandoc + XeLaTeX 的中文字体管道

`export_pdf.py` 是一条独立的五阶段线性管道，先把 Markdown 转成 Pandoc AST，再渲染成 LaTeX，最后调用 XeLaTeX 编译成 PDF——中文渲染靠 `CJKmainfont` 指定中文字体 ：

```bash
pandoc book.md -o book.pdf \
  --pdf-engine=xelatex \
  -V CJKmainfont="Noto Sans CJK SC" \
  -V geometry:margin=2.5cm \
  -V fontsize=12pt \
  --highlight-style=tango \
  --toc --toc-depth=2
```

依赖检查是 fail-fast 三档：

- **pandoc** → 没有就硬退出（必选）
- **XeLaTeX** → 没有就警告，中文会变方框（推荐）
- **fc-list 中文字体** → 没有就回退优先级列表（可选）

所以中文 PDF 出问题的排查顺序永远是：先 `fc-list :lang=zh` 看字体 → 再 `which xelatex` 看引擎 → 最后 `pandoc --version` 看转换工具 。

## 六、双仓库配合：huashu-bookwriter + md2book

这里有个容易踩的坑——**创作和排版最好分开**。ZeroxZhang 同时做了另一个仓库 `md2book`，分工非常清楚 ：

> **huashu-bookwriter 负责「写书」——生成结构化 Markdown 书稿**
> **md2book 负责「出书」——把 Markdown 排版成 PDF 电子书**

md2book 用 Node.js + Playwright + Chromium 渲染，支持三种主题：

- **minimal** → 极简科技风，白底黑白灰，适合技术文档
- **academic** → 经典学术风，暖色纸张、深红强调，适合理论著作
- **playful** → 活泼插画风，渐变彩色封面，适合入门科普

它还支持 GFM、代码高亮、callout 提示框、`[!INFO][!TIP][!WARNING][!DANGER]`，以及 YAML frontmatter（title/subtitle/author/version/date/description）。

推荐的完整 10 步流程 ：

```text
1. 装 huashu-bookwriter → 2. 选书籍类型 → 3. 生成 Markdown 书稿
4. 补 frontmatter → 5. 装 md2book → 6. 装 Node/Playwright/Chromium
7. --check 环境 → 8. minimal 主题先导出 → 9. 查封面/目录/中文/代码块/分页
10. --verbose 或 HTML 降级排查
```

## 七、它在 skill 家族里的生态位置

把前四个库串起来，整条链路就完整了：

| 库 | 正向/反向 | 输入 | 输出 |
|---|---|---|---|
| `colleague-skill` | 正向 | 人的数字痕迹 | 行为 + 人格 Skill |
| `book-to-skill` | 正向 | 书籍/文档 | 知识框架 Skill |
| `nuwa-skill` | 正向 | 名人公开材料 | 认知操作系统 Skill |
| **`huashu-bookwriter`** | **反向** | **书 + 作者痕迹** | **写书方法论 Skill** |

前面三个都是「把某个对象变成 Skill」，huashu-bookwriter 是倒过来的——**它从一个人的书和仓库里逆推出「他怎么写书」，再把这个方法论装回去生成新书**。本质上它和 `nuwa-skill` 最近，只不过 nuwa 抽的是「怎么想」，它抽的是「怎么写」。

而且它们可以组合：让 `huwa-bookwriter` 管写书流程 + `nuwa-skill/paul-graham` 管认知视角 + `book-to-skill/DDIA` 管知识框架 + `colleague-skill/你的同事` 管 Code Review 口味，就是一个完整的「AI 写作流水线」了。

## 八、使用边界：三点必须知道

1. **这是逆向工程版，不是花叔官方原始 Skill**——花叔本人的橙皮书制作 Skill 并未开源，ZeroxZhang 是从 4 本橙皮书 + 开源仓库逆出来的，还原度虽高但不是 100% 
2. **适合技术手册、橙皮书、快速指南；不适合小说、纯理论书、复杂插图设计书**——它的 DNA 就是花叔的技术写作风格，换个文体会水土不服 
3. **PDF 导出链两套并存**——仓库自带 `export_pdf.py`（pandoc + XeLaTeX 路线），但 md2book 走 Node.js + Playwright 路线，实测更推荐后者做最终排版，前者适合轻量导出 

---

所以 huashu-bookwriter 的原理可以压成一句话：**从花叔已出版的技术书和开源仓库里逆推出五层写作体系（信任建立→知识路径→风格 DNA→质量铁律→Agent 协议），封装成 Claude Code Skill，配合 md2book 排版，让 Agent 按「花叔式」方法论从 0 产出一本结构完整、风格一致、可 PDF 交付的技术书**。

要不要我接着把「huashu-bookwriter + nuwa-skill + book-to-skill + colleague-skill」四条线合成一个完整工作流——讲清楚怎么从 0 搭一个「选题→调研→撰写→QC→排版→发布」的 AI 写作工厂？


我发现了什么
这个过程比我想的有意思。拆开之后我发现了五层东西。

第一层：渐进式信任建立
花叔每本书、每章的开头都有时间线锚点。不是泛泛而谈的"我用了很久"，是具体的"用了 3 个月 Cursor 之后切换到 Claude Code"。

这个设计的核心是建立可信度。读者看到开头就知道——这个人真的用过这个东西，他不是拿文档在编。

逆向出来的 Skill 里保留了这个规则：每章前 2-3 段必须有具体时间线锚点或场景还原。没体验就诚实标注，不编造。

第二层：结构化的知识传递路径
三本书都遵循同一个递进逻辑：Part 1 从零到第一次跑通，Part 2 深入核心能力，Part 3 多场景实战。

这不是随便分的。每章解决一个具体问题，读者读完一章能做一件事。章节之间有向前桥接，不让读者停在某处不知道接下来该看什么。

这个递进路径是整套系统的骨架。

第三层：风格 DNA
这是我拆解过程中最有收获的部分。

花叔的写作不是"写得好看"，是写得精确。有几条规则我直接抄过来：

单句不超过 25 字，用句号断句

第一人称高频出现，每千字至少 3 次

具体数字，不用模糊量词

禁用词表：不用"综上所述"、"值得注意的是"、"接下来我们将"

有体验就断言，没体验就诚实标注

最让我惊讶的是禁用词表。不是随便写几条"不要水"，是明确的禁用词和替代词的对照表。比如"进行操作"换成"点击"或"输入"，"实现功能"换成"做到"或"搞定"。

这种精确度让我意识到——风格一致性不是靠天赋，是靠规则。

第四层：TDD 式质量铁律
这个最狠。

每章写完后要过 12 项 QC 检查：结构、风格、内容三块。全书写完再过 10 项全局 QC。

不通过就重写。不妥协。

更绝的是有个叫 Rationalization Table 的东西——专门对付 Agent 找借口跳过规则的。比如 Agent 会说"开头不重要，内容才是关键"，表里直接回："花叔风格的核心就是开头。没有锚点的章节不可信。"

这个设计太聪明了。它不是信任 AI，是给 AI 画线。

第五层：Agent Protocol
写作前必须先研究确认事实。版本信息、API 是否存在、社区反馈。不凭训练数据编造。

这一步保证了内容不是训练数据的幻觉，是真正经过验证的事实。
